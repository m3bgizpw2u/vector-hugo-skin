## [Unreleased]

### Fixed
- Article-body `{{< figure >}}` chrome no longer clashes in dark mode. `assets/css/components/figure.scss` hard-coded MediaWiki's light-mode hex literals (`background-color: #f8f9fa`, `border: 1px solid #a2a9b1`), so in dark mode the figure rendered as a bright light-gray box with a light border sitting on the dark page. Swapped both for the theme tokens `--color-surface-alt` and `--color-border`, whose `:root` (light) values are byte-identical to those hex codes — so light mode is pixel-unchanged, while dark mode now picks up the dark surface (`#27292d`) and border (`#72777d`) automatically through the existing `themes/dark.scss` cascade. One file touched.
- Floated `{{< figure >}}` image border wrapped a box roughly 2× the width of the rendered image, leaving a large empty background gap beside the picture. Root cause: a floated `.figure` shrink-wraps to content while the nested `.thumb`'s tablet tier used a *percentage* width (`clamp(12rem, 40%, 20rem)`), which is circular inside a shrink-wrap float — the browser resolved it against the image's intrinsic max-content instead of the rendered width (computed styles showed a 385px figure around a 192px image). Moved the width cap onto the `.figure` wrapper (whose containing block is the definite-width `.article-body`, so the clamp resolves cleanly) via `[data-kind="image"]` tiers mirroring `_thumb.scss`'s breakpoints (desktop `20rem`, tablet `clamp(12rem, 40%/50%, 20rem)`, mobile full-width), and made the nested `.thumb` fill it with `width: 100%` (selector `.figure > .thumb.figure__thumb`, specificity `0,3,0`, beats `_thumb.scss`'s `.thumb[data-align="none"]`). The border now hugs the image at every tier (verified live: 296px figure around a 288px image). Audio/video figures are untouched — the tiers are keyed to `[data-kind="image"]`. One file touched: `assets/css/components/figure.scss`.
- Paired-form `{{< figure >}}` caption rendered its Markdown as literal text and dropped the image. `layouts/_shortcodes/figure.html` treated any non-empty `.Inner` as a full body **replacement** rendered through `safeHTML` (no Markdown parsing) and skipped the image element entirely, so `{{< figure src="…" >}}Caption with a [link](https://…).{{</ figure >}}` showed the raw `[link](url)` text with no picture. Aligned the shortcode with the sibling `{{< thumb >}}` convention: a non-empty paired body is now the **caption**, rendered as Markdown via `.Page.RenderString` (inline links / emphasis work) and taking precedence over the `caption=` param, while the image / audio / video element is always emitted from the parameters. Docblock updated to describe the paired-body-as-caption semantics. One file touched: `layouts/_shortcodes/figure.html`.
